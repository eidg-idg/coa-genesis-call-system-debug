🎯 Initiative Summary
What This Work Is

Colorado Access uses the EPlus managed package (Engagement Plus, v23.1.0.0) as their primary member and provider verification tool inside Salesforce Service Center. When a Genesys phone interaction is delivered to an agent, EIDG's custom Genesys screen pop LWC components launch alongside the EPlus verification workflow.

Body 1 work (completed) addressed save reliability, caller identity capture, and ANI pass-through. Body 2 addresses conditional rendering parity — making the Genesys screen pop LWCs match EPlus's field-by-field display rules so CSRs see a consistent, accurate interface across both systems.

🔬 How These Findings Were Produced
📸Source 1: Client-supplied Member Provider Search.docx — 10 screenshots (5 EPlus / 5 Genesys side-by-side, Member Non-Member path)
🖥️Source 2: Live sandbox screenshots — all EPlus modal states for both Member and Provider workflows (all Case Origins, all Representative Types, "on behalf" checked/unchecked)
🔒Constraint: EPlus is a managed package — source code is inaccessible. All findings are based on direct UI observation. EPlus is the definitive reference standard.
✅Confidence: Both workflows fully documented. 4 minor sandbox items still pending (Verify behavior, Caller Type options).
⚡ Key Findings at a Glance
🔴Critical Member bug: Non-phone call origins (Email, Walk-In, Research, etc.) cannot complete verification in the current LWC. EPlus allows immediate Verify — our LWC blocks them entirely.
🔴Critical Provider bug: Provider verification checkboxes are always shown regardless of channel. EPlus only shows them for phone calls — non-phone should show Case Origin + Verify only.
🔴Data integrity risk: 3 Provider Case Origin values are stored incorrectly. Reports filtering on "Voice Mail" or meeting origins will miss records saved by our LWC.
🟢EIDG additions are sound: Caller Name, Caller Phone (ANI-prefilled), and Provider Status are intentional EIDG enhancements beyond EPlus. Most should remain pending COA decisions.

🔴 Critical Bug — M-P1-02: Non-Phone Origins Cannot Complete Verification

In the current Genesys LWC, the verification function requires both a Case Origin value and a Member Type value before proceeding. EPlus only shows Member Type for Inbound and Outbound Phone Calls.

This means a CSR selecting Chat, Walk-In, Research, Transfer, Email, Fax, Voice Mail, Mail, or Meeting – Virtual can never advance to complete verification in the current LWC. This is a production bug.

Fix: For non-phone origins, allow Verify to proceed on Case Origin selection alone — no Member Type required.

✅Caller Name (required, always visible) — Not in EPlus. Ensures caller identity is always captured regardless of interaction type.
✅Caller Phone (optional, ANI pre-filled from Genesys) — Not in EPlus. Reduces manual data entry; ANI auto-populates from the screen pop payload.

▶M-P1-01 Case Origin shown first, alone. Member Type only appears after selection (phone origins only). Case Origin + Member Type shown simultaneously from modal open. Flow sequencingMedium
The Gap
EPlus guides the CSR through two steps in sequence: first select a Case Origin, then — only if it's a phone call — select a Member Type. Our LWC presents both dropdowns simultaneously from the moment the modal opens.
Why It Matters
Showing Member Type upfront is unnecessary and slightly misleading for non-phone interactions, since EPlus never asks for it in those cases. A CSR familiar with EPlus will notice the layout looks different from what they expect.
What Needs to Change
Show Member Type only after Case Origin has been selected and only when the selected origin is a phone call. For non-phone origins, Member Type should never appear.
▶M-P1-02 Non-phone origins: Verify available immediately. No Member Type required. Non-phone paths can never complete verification — code requires both fields. Critical bugHigh
The Gap
In EPlus, selecting a non-phone Case Origin (such as Email, Walk-In, or Research) immediately shows a Verify button — no Member Type selection is needed. In our LWC, the underlying code requires both a Case Origin value and a Member Type value before it unlocks the verification section. Since Member Type only appears for phone origins, non-phone CSRs are stuck — the verification section never appears for them.
Why It Matters
This is a production bug. Any CSR attempting to verify a member through a non-phone channel using the Genesys screen pop cannot complete the verification. The modal will appear functional but will never advance past the initial fields.
What Needs to Change
For non-phone Case Origins, the code must allow Verify to proceed based on Case Origin alone — without requiring a Member Type selection. Member Type should only gate the flow for phone call origins.

Update the button label and add a left-arrow icon to match EPlus's "← Go Back" styling.

The Gap
EPlus shows no Caller Name field for non-phone Case Origins (Email, Voice Mail, Research, Meetings). Our LWC shows Caller Name at the top level for all origins and the underlying code requires it before Verify can succeed. Once P-01 is fixed and the checkbox section is gated behind phone origins, a CSR doing an Email verification would still see a required Caller Name field with no way to dismiss it — blocking Verify.
Why It Matters
This is a secondary bug that will be created by fixing P-01. It must be resolved at the same time, or the P-01 fix will trade one broken non-phone path for another. For non-phone interactions (Email, Research, etc.), there may genuinely be no "caller" — requiring a name entry is awkward and inconsistent.
What Needs to Change
Depends on Client Decision CD-01. The options are: (A) gate Caller Name behind phone origins to fully match EPlus; (B) keep Caller Name visible but optional for non-phone origins; or (C) keep it required for all origins and explicitly diverge from EPlus. EIDG recommends Option A.
🟣 Client Decisions Required

The following three decisions require COA input before implementation can proceed. Each is independent and can be discussed in any order.

CD-01 · Provider Verification
Should "Caller Name" be captured for non-phone interactions?
🔴 Required before Provider work begins

When a CSR verifies a provider, they first select a Case Origin. Phone call origins (Inbound, Outbound) show the full verification layout including caller information fields. Non-phone origins (Email, Voice Mail, Research, Meeting – Virtual, Meeting – In Person) show only Case Origin in EPlus — no checkboxes, no caller information.

Our current Genesys screen pop (Body 1) added a "Caller Name" field that is always visible and always required — even for Email, Voice Mail, and Research interactions where there may be no live caller. Once the phone/non-phone gate is fixed (P-01), this required field will block non-phone verifications unless we also adjust it.

Note: This decision is Provider-specific. The Member modal resolves this naturally — non-phone Member verifications bypass the screen where Caller Name lives entirely.

Option A — Full Parity EIDG Recommends✓
Gate Caller Name and Phone behind phone origins. Non-phone verifications capture Case Origin only — no caller identity fields shown. Matches EPlus and is consistent with Member behavior.
Option B — Optional Capture✓
Keep Caller Name visible for all origins but make it not required for non-phone. CSR can enter it if relevant, but is not blocked. Slight UX ambiguity when there is no live caller.
Option C — Always Required✓
Keep current behavior — Caller Name required for every verification regardless of channel. Maximizes data capture. May be awkward for Email, Voice Mail, or Research workflows.
CD-02 · Provider Verification
Should the "Provider Status" checkbox remain?
🔴 Can be decided independently

Our Genesys screen pop includes a "Provider Status" checkbox in the provider verification list. EPlus does not have this checkbox — it is not part of the standard EPlus provider verification workflow.

Unlike Caller Name and Caller Phone (which are explicitly documented Body 1 additions), Provider Status does not appear in the Body 1 change log. It is unclear whether it was intentionally added or is a remnant of earlier development.

Important prerequisite: Before deciding, EIDG should confirm whether the Salesforce verification record object has a field to store a Provider Status value. If no storage field exists, the checkbox cannot save data and should be removed regardless.

Option A — Remove✓
Remove Provider Status for full EPlus parity. Recommended if no underlying storage field exists on the verification record.
Option B — Keep as EIDG Add If field exists✓
Treat as an intentional EIDG enhancement for enhanced data capture. Document formally alongside Caller Name/Phone. Requires confirming the field exists and data saves correctly.
CD-03 · Member Verification
How should authorized representative information be captured?
🟡 Lower urgency — affects Body 2 scope significantly

When a caller identifies as a Non-Member (calling on behalf of a member), EPlus asks for Representative Type. Depending on the type selected, EPlus either searches existing authorized representative records on file (Legal Rep, Parent) or lets the CSR create a new authorization record in real time (Guardian, POA, Advocate, etc.) including name, phone, start date, and end date.

Our current Genesys screen pop shows only a free-text Description textarea for all representative types. It does not access EPlus records and does not create new authorizations.

Key question for COA: Does representative data entered through the Genesys screen pop need to feed back into the EPlus managed package records — or is it supplementary data captured only on the Salesforce verification record?

Option A — Full EPlus Parity✓
Build lookups for existing EPlus auth rep records; build a form to create new records during verification. Requires access to managed package objects. High complexity.
Option B — Partial Match✓
Show Name and Caller Phone as free-text inputs (not lookups) for Guardian/POA/etc. Captures the same data points without requiring managed record creation. Medium complexity.
Option C — Keep Description Pending COA input✓
Keep current Description textarea as supplementary notes. Simple, no managed package dependency. Best if rep data does not need to sync back to EPlus. No change required.
